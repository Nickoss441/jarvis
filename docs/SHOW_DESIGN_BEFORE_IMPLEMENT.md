# Show Design Before Implementing

**Purpose:**
User wants to see/approve the plan or design before any code is written.

## Workflow
1. For any UI, feature, or non-trivial change, write out a clear description or sketch of the intended design, layout, or logic.
2. Present this plan to the user for review.
3. Wait for explicit approval before making any code or file changes.

**Why:**
The user explicitly said "I want to see it before you implement it" — they don't want surprise implementations.

**How to apply:**
- For UI: Describe layout, colors, interactions, and flow.
- For features: Outline logic, endpoints, and data flow.
- For backend/infra: Summarize architecture, file changes, and integration points.

Always ask: “Want me to go ahead?” before implementing.
